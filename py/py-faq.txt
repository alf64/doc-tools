---------------------------- Basics ----------------------------------
Hierarchy:
 1. Module - a source file of .py extension. Contains source code (set of functions).
 2. Package - a set of modules.
 3. Library - a set of packages.

Documentation:
For the built-in python functions and packages, there is a nice html documentation provided in the place
on your computer where the python has been installed, within the Docs/ directory. Find the index.html
file there and run it.

-------------- Python Launcher - Selecting python version, listing available pythons --------------------
If you have more than one version of python installed, use python launcher ('py') to select the one you want.
# See help of python launcher
py --help
# List installed pythons. Note that the '*' indicates the default python used when no version is specified during a python call.
py --list
# Or list them using following command, which will also provide their installation directories
py -0p
# Call your python script with python 2 (assuming python 2 is installed)
py -2 <your-python-script>
# Call your python script with python 3 (assuming python 3 is installed)
py -3 <your-python-script>
# Call your python script with particular version of python (in this case: 3.9 - assuming you have it installed)
py -3.9 <your-python-script>

-------------- virtual environment (venv) --------------------
1. Create virtual environment
py -m venv <name-of-venv>
The directory wth name <name-of-venv> will be created and initial virtual environment files will be created there.

2. Activate virtual environment
cd <name-of-venv>\Scripts
activate.bat
Now in the console the name of the virtual env will be displayed near the prompt.

3. Deactivate virtual environment
cd <name-of-venv>\Scripts
deactivate.bat

4. Info about usage of virtual env
When you have the virtual env activated, please call python from within it.
I.e.: 
<your-virtual-env-name>\Scripts\python.exe <name-of-the-py-script-to-call>
This should assure that you use the python and packages from within virtual env.
It would be good to use pip also from this location when installing or uninstalling packages.
Installed packages shall fall into:
<your-virtual-env-name>\Lib\site-packages
TIP: <your-virtual-env-name>\pyvenv.cfg holds info from what python version the virtual env has been created

4. Moving virtual env between different computers
Upon moving the virtual env to a new computer or new location, you have to edit some paths within it.
Otherwise it will try to use its previous location and you may encounter error like 'ModuleNotFound' and so on.
The following has to be done after moving to a new location:
- delete <your-virtual-env-name>\Lib\site-packages\__pycache__ directory
- edit the <your-virtual-env-name>\Scripts\activate - update paths that will match the new location of your virtual env
- edit the <your-virtual-env-name>\Scripts\activate.bat - update paths that will match the new location of your virtual env
- edit the <your-virtual-env-name>\Scripts\jp.py - update paths that will match the new location of your virtual env

-------------- Managing packages --------------------
# package manager is called pip and you can ask it for help
pip --help

# list installed packages (including all, even pip)
pip list

# list installed packages (excluding things like pip)
pip freeze

# dump list of installed packages into txt file
pip freeze > requirements.txt

#  Install package of a given <package-name>
pip install <package-name>

#  Install package of a given <package-name> and particular version
pip install <package-name>==<desired-package-version>

# Install packages listed in txt file
pip install -r requirements.txt

#  Uninstall package of a given <package-name>
pip uninstall <package-name>
# it does not uninstall dependencies

# Install packages listed in txt file
pip uinstall -r requirements.txt

# locate where the package has been installed on the system
pip show <package-name>

# download (without installing) package into a folder. It downloads all dependencies also.
pip download <packagename> -d path_to_folder

# install package from a folder
pip install --no-index --find-links=path_to_folder <packagename>

-------------- Exporting packages from one PC ('source', which is online) to import them into another PC ('dest', which is offline) --------------------
# On the 'source' PC, dump the list of installed packages into the text file
pip freeze > requirements.txt

# On the 'source' PC, download (from the web through pip) the packages from the list into a folder.
# The 'download' sub-command only downloads the packages, but does not install them. It downloads the dependencies also.
pip download -r requirements.txt -d path_to_the_folder

# Move the folder with packages onto the 'dest' PC. Move the requirements.txt file also.

# On the 'dest' PC run the following command that will install all packages mention in requirements.txt:
pip install -r requirements.txt --no-index --find-links=path_to_the_folder
# Alternatively you can install single particular package manually with the following command:
pip install --no-index --find-links=path_to_the_folder <packagename>

-------------- Using pyinstaller to bundle script and dependencies into single .exe --------------------
# install pyinstaller
pip install pyinstaller
# once installed, pyinstaller can be called through cmd directly just like pip

# Create .exe from your script and set icon based on .ico file:
pyinstaller --onefile mainScript.py
# pyinstaller parses your mainScript.py and resolves all dependencies (sub-scripts, packages, binaries)

# pyinstaller options:
--windowed                 - means there will be no console involved for I/O at all. You won't see anything in cmd.
                           Use this if your script is a pure GUI one.
--name=nameForYourExe      - name of the .exe file. If you don't pass this, the name will be equal to script name.
--icon=icon.ico            - icon file to be used for .exe. If you don't pass this, default one will be used.

# pyinstaller creates:
 - dist/                   - directory with created .exe file
 - build/                  - directory that is temporary for building .exe and you can delete it
 - .spec                   - file with options that pyinstaller used to create .exe
                           It can be deleted after creating .exe, but it may be cool to keep it
                           because it can be re-used to once again call pyinstaller with the same options.
                           To call pyinstaller using these options: pyinstaller options <.spec file>
